Spring boot microservice example with StackSAGA

Overview

in this section, we are going ot move to the StackSaga work flow to get the features the framework provides. StackSaga provides a smooth layered-architecture to overcome the Saga Execution Coordinator (SEC) capabilities the framework provides.

The time has come to move to the design pattern. here you can see the different between the typical MVC architecture and the CHES architecture the StackSaga provides. there is no big difference there. only thing is that additionally 2 layers will be added in your layers flow called Handler layer and Executor layer between the controller layer and the service layer. those invocations of the layers are taken care of by the framework.

Adding StackSaga dependencies

To use the StackSaga, you have to add the StackSaga dependencies to in your pom.xml file. here you have to add 2 dependencies. 1st one is the core-dependency and 2nd one is the implementation of the database accessing dependency. you can choose it as per your requirement. if you are willing to use Mysql database as your event-store, you can use stack-saga-spring-boot-starter-mysql dependency, or if you are willing to use MongoDb as your event-store, you can use stack-saga-spring-boot-starter-mongo dependency. See more available implementation dependencies. in this example, we use Mysql database as the event-store database.



Creating the Aggregator class for place order process.

we have to create a common class object for carrying the data through the executors. the class is not just a class. it will be the base key for all process reading the place-order process. The class contains all the data fields that you want for each execution like make-payment, increase-point and all others.

The created class should be extended from the SagaAggregate class. and it should be annotated as @Aggregator. the annotations contain a required property called version. it is the location where you manage the version of the whole place-order process just like managing the rest endpoint version. if you update something of the old process, the next version will be 1.0.1.

create your PlaceOrderAggregator class inside a package called aggregator as a best practice.
then all aggregator classes in the project will be stored under the aggregator package.

order-service > org/mono/stacksaga/example/orderservice/aggregator/PlaceOrderAggregator.java

Identifying The executors for each operations

# Service Executor Name Executor Type Process-Transaction Compensation-Transaction
1 order-service POCreateOrderExecutor COMMAND Executor ORDER_CREATED ORDER_CANCELED
2 user-service POCheckUserExecutor QUERY Executor CHECK_USER -
3 payment-service POMakePaymentExecutor COMMAND Executor MAKE_PAYMENT REFUND_PAYMENT
4 user-service POIncreasePointsExecutor COMMAND Executor INCREMENT_POINTS DECREMENT_POINTS
5 delivery-service PODispatchExecutor COMMAND Executor ADD_TO_DELIVERY CANCEL_DELIVERY

According to the summarized table, you can see there are two types of executors called command-executors and query-executors are there. and all the command executors can have 2 method to be invoked. and all the query executors can have one method to be executed.

By now we have all the services for execute the sub operations of the place-order process like create-order, make-payment, increase-points ect. this is the time to create executors by using the services that we created earlier.

Creating The executors for each operations



POCreateOrderExecutor

POCreateOrderExecutor is the relevant executor for the create-order and its compensation. due to the executor is a command executor, the POCreateOrderExecutor class must be implemented by the CommandExecutor<Aggregator> by passing the PlaceOrderAggregator class that you created as the type. and then you have to implement two methods coming from the CommandExecutor interface. those are the methods for execute the process-transaction doProcess() and the compensation-transaction doRevert(). and after that the class should be annotated as a @Executor. then spring will create a bean for you. the annotation has a required property called executeFor. you have to mention what is the name of the service that you are going to access by using this executor class. the service name should be same as the spring.application.name. upon that the framework identify what services accessed by this service and then this base server will know about the health of accessing services that you mentioned. read more about how retries does work. for POCreateOrderExecutor, the service name is order-service (own service name)
