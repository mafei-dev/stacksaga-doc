StackSaga in High-level

in the introduction, we got a clear idea of how microservice architecture works and what are the challenges that we have to face when implementing the microservice. here, we are going to explain how the stack saga works together with typical microservice architecture. here we have demonstrated a spring boot microservice architecture and how the StackSaga framework provides the Saga execution coordinator (SEC) capabilities without bothering to the default architecture.

Stacksaga is not just a framework that provides SCE facilities. Stacksaga has many features. The following features StackSaga provides.

  • Saga execution coordinator features.
  • Real-time circuit breaker features.
  • Distributed tracing dashboard.
  • Service's heath matrices monitoring dashboard.
  • Event sourcing
  • Self retryable execution

After adding stack saga in the microservice architecture you can see there are some additional components in the high level architecture diagram. Let's discuss all components and how they are interacted with general architecture. When you use StackSaga, you have to provide the facilities to create a database in addition to the database that you use for each microservice. This also same as the database per service design pattern. A service group (multiple instances of one service) uses a database to store the events of the executions. The only thing is newly added an extra schema. If you like to use one database for all microservices for StackSaga, it doesn't matter you can use one database. But as a best practice, database per service is recommended. After the database part, you can see a component called Redis. Redis is an in-memory data structure store, used as a distributed, in-memory keyâ€“value database, cache and message broker, with optional durability. In the framework architecture, Redis is used for communicating with each microservice and admin service and storing each service's live health data. What are the responsibilities of Redis in Stacksaga?

  • Collecting and Update the instance health for interested services.

    After adding the StackSaga dependencies into the application, the application called as StackSaga client application. That client application contains a configurable scheduler to update the health to the Redis service. Redis service save the health details in-memory and Redis push about server changes to other interested microservices and the admin service. See more how the CB works.

  • Work as a data transfer protocol thorough the StackSaga client and StackSaga Admin server.

    The framework communicate with all the microservices based with the admin service. Admin server retrieves some data from each microservices. But the framework doesn't user Http protocol to their communications. All the data is communicated through the Redis service.
