Configuration Properties Of StackSaga Starter Core

version 1.0.0

Name Type Default Description Since
stacksaga.component-scan java.lang.String [] no-default The package paths array where the executors, handles and other beans and aggregators have been configured. at start-up the configured paths will be scaned to find the beans and other implementations that you have done in StackSaga framework. 1.0.0
stacksaga.app-release-version java.lang.String no-default The real release version of the service. it should be updated for each and every change. especially regarding the Aggregators. (if any aggregator was changed, the version should be updated.) 1.0.0
stacksaga.log.configured-details java.lang.Boolean true you can see the configured data summary of the StackSaga framework like Executors, Aggregators and Etc. 1.0.0
stacksaga.aggregator.print-summary java.util.Map<java.lang.String,java.lang.Boolean> empty if you want to print aggregator summary st the service start-up, you can mention the aggregator name (@Aggregator(name= "TheNameOfTheAggregator")) and true or false. this will be helpfull when the development profile. you can disable other all other aggregator's summaries, and you can enable the aggregator summary you are currently working on. 1.0.0
stacksaga.log.transaction-summary java.lang.Boolean true you can see the transaction summary for each transaction in the log as quite beautiful table. it is not recommended in the production profile. 1.0.0
stacksaga.client.binary-file-path java.lang.Long no-default the file root-path to save the event binary files (garbage transaction files). if the service is running on a container-based system, you can provide a volume location for the service group. (no need to provide the full path with the service name. the framework will create the directory. for instance, the service name is order-service and the root-path is C:\\stacksaga\\binary and the framworl will create the full path like this C:\\stacksaga\\binary\\order-service). 1.0.0
stacksaga.client.trash-bulk-size java.lang.Long 10_000L by using the stacksaga.client.binary-file-path that provided, the coordinator tries to collect the garbage binary file from the directory. this value is belonged to that how many binary files should be collected and execute by the service.
this value should be decided upon the resilience (service resources) of the services because it consumes some resources to process the IO execution. 		1.0.0
stacksaga.cloud.file-collector-cron java.lang.String */1 * * * * * Schedule that how the garbage file-collector should be invoked by using cron values. 1.0.0
stacksaga.client.zone-id java.lang.String default-zone zone-id is usaed for determining the master node of the service group. see more 1.0.0
stacksaga.client.replay-bulk-size java.lang.Long 10_000L the bulk size of retreating retryable-transaction from the event-store.
when the coordinator tries to execute the retryable-transaction, the coordinator will fetch the retryable-transaction transaction from the event-store. this value means the limit of the database select query. (how many data should be fetched per one time.)
the bulk size should be decided upon the resilience of the services. because anyone does not know which service will act as the master-node. master node is responsible for executing the bulk spray,and one of same services will be a master node based on the time that server was started. 		1.0.0
stacksaga.client.replay-bulk-order-by

java.lang.String

 ASC It describes that what order the retryable-transaction execution should be done. ASC or DESC.
if the value is ASC, first-in > first-out,
if the value is DESC, last-in > first-out 		1.0.0
stacksaga.cloud.transaction-replay-cron java.lang.String */1 * * * * * Schedule that how the retryable-transaction invoker should be invoked by using cron values. 1.0.0
stacksaga.connect.admin-url java.lang.String no-default the full url of the admin-server running on. 1.0.0
stacksaga.connect.admin-username java.lang.String no-default the username of the user account that has been created for specifically to the relevant service by the SUPER-ADMIN or ADMIN. see more.. 1.0.0
stacksaga.connect.admin-password java.lang.String no-default the password of the user account that has been created for specifically to the relevant service by the SUPER-ADMIN or ADMIN. see more.. 1.0.0
stacksaga.connect.enable java.lang.Boolean true should the service connect with the StackSaga admin server at the start-up or not.
the benefit of enabling is you can see all the instance details through the StackSaga admin server like what time the service was started, what was the instance-id Etc. (equal to the eureka that you can see). if the value is true, you should create a account with SERVICE privileges in the StackSaga admin server. 		1.0.0
stacksaga.executor.executor-retry-global-config java.lang.Integer no-default The default value for that how many hours should be waiting a retryable transaction until get successful. (if there is a transaction as a retryable transaction, it can try until the time that you have mentioned here by hours.) for instance, you have mentioned the value as 12 hours. that transaction can be run for 12 hours to be successful. otherwise the transaction doesn't try to invoke again and again. 1.0.0
stacksaga.executor.executor-retry-config java.util.Map<String,ExecutionRetry> empty if you want to configure retry-hours individually for selected aggregators instead of using global value (stacksaga.executor.executor-retry-global-config), you can proce the data by using the aggregator name that mentioned at the @Aggregator(name= "TheNameOfTheAggregator")
example,
stacksaga.executor.executor-retry-config.OrderAggregator.hours=20 		1.0.0
